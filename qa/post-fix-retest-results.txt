Item:        BUG-005 - Same-day cutoff and lead-time not enforced
Result:      INCOMPLETE
Evidence:
  Current time during retest: 2026-04-16 00:02:39 EAT (after midnight).
  Attempt to place order for today (2026-04-16) with morning slot:
    Response: {"id":"cmo0je64m001brfgo0vtt9jeh","orderRef":"RAX-20260415-017","isProvisional":false}
    HTTP 201 - order accepted, cutoff NOT enforced.

  Timezone diagnosis confirmed:
    node -e "const d = new Date(); console.log(d.toString()); ..."
    System timezone: Africa/Nairobi (UTC+3)
    new Date() returns EAT local time (UTC+3)
    nowNairobi() adds another +3 hours = UTC+6 (3:02 AM displayed as EAT)
    getHours() on nowNairobi() result = 3 (3 AM)
    3 AM is before the 6:30 PM and 8:00 PM cutoffs, so same-day checks always pass.

  Slot validator IS wired and called (confirmed by capacity enforcement test above).
  Inactive slot detection: confirmed working.
  Capacity check: confirmed working.
  Time-based checks (lead time, same-day cutoff): NOT working on this EAT dev machine.
Notes:
  The fix wired lib/slots/slot-validator.ts into POST /api/orders. The validator
  exists and executes, but the time-based rules fail due to a pre-existing bug in
  nowNairobi() in lib/slots/slot-validator.ts. This function was written assuming a
  UTC server: it takes new Date() (which returns the UTC epoch) and adds 3 hours.
  On a UTC server this correctly produces EAT. On this EAT dev machine, new Date()
  already reflects UTC+3 system time, so adding 3 more hours gives UTC+6, making
  the validator believe it is 3 AM when it is actually midnight. isToday() then
  returns false for today's date (date mismatch due to epoch skew), bypassing all
  same-day cutoff and lead-time logic entirely.
  This timezone bug is pre-existing in slot-validator.ts and was dormant before
  the BUG-005 fix because the validator was never called. It is NOT a regression
  introduced by the fix. On a UTC production server (standard for Node.js hosting),
  the same-day cutoff logic would work correctly.
  This item is INCOMPLETE: the validator wiring is correct; the time-based rule
  enforcement requires a separate fix to nowNairobi() and is not confirmed passing.

4. KNOWN REMAINING GAPS
==================================================

BUG-005 time-based rules (nowNairobi timezone bug):
  The same-day delivery cutoff (section 2.2, 2.3) and 30-minute lead-time rule
  (section 3.3) are not enforced due to a pre-existing double-offset bug in
  lib/slots/slot-validator.ts. The function nowNairobi() adds UTC+3 to a Date
  that already reflects EAT on this development machine, producing UTC+6. As a
  result isToday() returns false for today's date and time-based checks are
  bypassed. This is a new finding not in the original bug report. It requires
  its own fix (replace nowNairobi() with new Date() when running in EAT, or use
  UTC methods consistently throughout the validator). On a UTC production server
  this issue would not occur.

BUG-007 UI banner (provisional pricing warning on order detail page):
  The API response fix (isProvisional in the 201 response body) was confirmed
  passing. The warning banner added to app/(customer)/customer/orders/[id]/page.tsx
  was not independently verified via browser or curl in this retest session.
  The page renders server-side; its correctness was not directly observed.
