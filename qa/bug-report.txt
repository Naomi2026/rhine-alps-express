2. RELEASE READINESS
══════════════════════════════════════════

Verdict: INTERNAL DEMO ONLY

Key reasons:
  1. Notifications are completely non-functional due to a foreign key mismatch bug.
     Customers receive no order confirmations and no OTP codes. In production this
     means no customer would know their delivery OTP, making delivery confirmation
     impossible without manual admin intervention.
  2. Slot capacity enforcement (§3.4) is completely absent. SlotCapacity.bookedOrders
     is never incremented or decremented. Overbooking cannot be detected or prevented.
     The same-day delivery cutoff and 30-minute lead-time rules are not enforced at
     checkout, meaning customers can book a 9 AM slot at 11 PM.
  3. The API returns 307 HTML redirects for unauthenticated requests instead of 401 JSON.
     Any non-browser client (mobile app, API integration) will receive an unparseable
     response on auth failure.

Minimum required before any user-facing release:
  - Fix BUG-001: Correct the notification service to pass User.id (customer.userId)
    instead of Customer.id (customer.id) so OTP codes reach customers.
  - Fix BUG-002 + BUG-007: Wire lib/slots/slot-validator.ts into POST /api/orders
    to enforce capacity limits and the same-day cutoff and lead-time rules.
  - Fix BUG-003: Return JSON 401 responses from API routes when unauthenticated,
    not HTML redirects.
  - Replace notification stubs with at least one working channel (SMS or in-app push)
    so OTP codes can actually be delivered to customers.

Bug ID:             BUG-001
Title:              Notification service passes Customer.id instead of User.id — all notifications silently fail
Severity:           High
Area:               Notifications
Status:             Open
Environment:        Local QA
Reproducible:       Always

Steps to Reproduce:
  1. Sign in as any customer (e.g. jane.kamau@example.com / Customer123!)
  2. POST http://localhost:3000/api/orders with valid order body:
       {"deliveryAddress":"Near dispatch","deliveryLat":-1.145,"deliveryLng":36.961,
        "slotId":"<any active slot id>","deliveryDate":"2026-04-20","quantity":1,
        "isExchange":false,"paymentMethod":"MPESA"}
  3. Observe HTTP 201 response — order is created successfully.
  4. Run: SELECT count(*) FROM "Notification";
  5. Check server logs for error output.

Expected Result:
  Two Notification records are created in the database:
    - event: order.placed, linked to the customer's user account
    - event: otp.sent, linked to the customer's user account
  Per business-rules.md §20.1: "Order placed confirmation shall be sent to the customer."
  Per §15.1: OTP notification shall follow the configured behavior.

Actual Result:
  Zero Notification records created.
  Server logs contain:
    [notification-service] Failed to send notification: PrismaClientKnownRequestError:
    Invalid `prisma.notification.create()` invocation:
    Foreign key constraint violated on the constraint: `Notification_userId_fkey`
    at lib/notifications/notification-service.ts:51

Evidence:
  SELECT count(*) FROM "Notification";
   count
  -------
       0
  (1 row)
  (After placing 5 orders during QA session — zero notifications in DB)

  Server log excerpt:
    prisma:error
    Invalid `prisma.notification.create()` invocation:
    Foreign key constraint violated on the constraint: `Notification_userId_fkey`
    [notification-service] Failed to send notification: Error [PrismaClientKnownRequestError]
    at async send (lib/notifications/notification-service.ts:51:26)
    at async notifyOTPSent (lib/notifications/notification-service.ts:171:3)

Likely Root Cause:
  In app/api/orders/route.ts, the notification call passes customer.id (the Customer
  table's primary key) as the customerId:
    notifyOrderPlaced({ customerId: customer.id, ... })
  The notification service then forwards this value as userId to db.notification.create():
    await db.notification.create({ data: { userId: data.customerId, ... } })
  But Notification.userId is a foreign key to User.id — a different ID from Customer.id.
  The two are never equal. The insert fails with a FK violation on every attempt.
  The .catch(console.error) pattern on the fire-and-forget call suppresses the error
  from the API response, so orders complete successfully but notifications are silently lost.

Likely Files Involved:
  app/api/orders/route.ts (lines ~110-130: notification calls pass customer.id)
  lib/notifications/notification-service.ts (notifyOrderPlaced, notifyOTPSent,
    resolveChannel — all receive Customer.id, need User.id)

Recommended Fix:
  In app/api/orders/route.ts, change the customerId passed to notification helpers
  from customer.id to customer.userId. Specifically:
    notifyOrderPlaced({ customerId: customer.userId, ... })
    notifyOTPSent({ customerId: customer.userId, ... })
  Apply the same correction to any other notification calls that currently receive
  a Customer record's id field. The notification service parameter naming should
  also be updated to reflect that it expects a User.id, not a Customer.id, to
  prevent recurrence.

Bug ID:             BUG-002
Title:              Slot capacity (SlotCapacity.bookedOrders) never incremented — overbooking undetectable
Severity:           High
Area:               Slots
Status:             Open
Environment:        Local QA
Reproducible:       Always

Steps to Reproduce:
  1. Note the active slot ID from: SELECT id FROM "DeliverySlot" WHERE "isActive"=true LIMIT 1;
  2. Place 5 or more orders for the same slot, same date, same pricing band using any customer session.
  3. Run:
       SELECT "bookedOrders", count(*) FROM "SlotCapacity" GROUP BY "bookedOrders";
  4. Also run:
       SELECT sc."bookedOrders", COUNT(o.id) as actual_orders
       FROM "SlotCapacity" sc
       LEFT JOIN "Order" o ON o."slotId"=sc."slotId"
         AND o."deliveryDate"=sc.date
         AND o.status NOT IN ('CANCELLED','COMPLETED')
       GROUP BY sc."bookedOrders"
       HAVING sc."bookedOrders" != COUNT(o.id);

Expected Result:
  Per business-rules.md §3.4: "Slot capacity shall be measured as the maximum number
  of orders allowed per zone per delivery slot." bookedOrders should increment each time
  an order is placed for that slot/band/date, and decrement on cancellation. Orders
  exceeding maxOrders (default: 20) should be rejected or redirected to the next slot.

Actual Result:
  bookedOrders = 0 for all 12 SlotCapacity rows regardless of how many orders are placed.
  8 orders were placed during the QA session; no capacity was ever recorded.
  The discrepancy query returns 3 rows showing actual order counts vs 0 capacity tracked.

Evidence:
  SELECT "bookedOrders", count(*) FROM "SlotCapacity" GROUP BY "bookedOrders";
   bookedOrders | count
  --------------+-------
              0 |    12
  (After 8 orders placed across multiple slots)

  Capacity discrepancy check:
   slotId                    | pricingBand | date       | capacity_booked | actual_orders
   cmo07pjft000krftem88bxmuq | BAND_1      | 2026-04-14 |               0 |             1
   cmo07pjfu000lrftewtaqw9m0 | BAND_2      | 2026-04-14 |               0 |             1
   cmo07pjfu000mrftele4qxy51 | BAND_3      | 2026-04-14 |               0 |             1

Likely Root Cause:
  lib/slots/slot-validator.ts implements full capacity validation logic (checkSlotAvailability,
  suggestNextSlot, meetsLeadTime, checkSameDayEligibility) and was built in Phase 2.
  However, app/api/orders/route.ts only checks slot.isActive — it never imports or calls
  anything from slot-validator.ts. SlotCapacity records are never read during order placement
  and bookedOrders is never incremented. The validator is dead code as far as the order
  flow is concerned.

Likely Files Involved:
  app/api/orders/route.ts (slot check at line ~80: only checks slot.isActive)
  lib/slots/slot-validator.ts (fully implemented, never imported or called)

Recommended Fix:
  In app/api/orders/route.ts, after confirming the slot is active:
  1. Import checkSlotAvailability (or the relevant functions) from lib/slots/slot-validator.ts.
  2. Call the validator with the slot, deliveryDate, pricingBand, and capacity records.
  3. If validation fails (capacity full, past cutoff, insufficient lead time), return 400
     with the reason and optionally suggest the next available slot.
  4. On successful order creation, increment SlotCapacity.bookedOrders for the matching
     slotId + pricingBand + date row (create the row if it does not exist for that date).
  5. In the cancel route (app/api/orders/:id/cancel/route.ts), decrement bookedOrders
     when an order is cancelled.

Bug ID:             BUG-003
Title:              Unauthenticated API requests return 307 HTML redirect instead of 401 JSON
Severity:           Medium
Area:               API Validation / Auth
Status:             Open
Environment:        Local QA
Reproducible:       Always

Steps to Reproduce:
  1. Without any session cookie, call any protected API route:
       curl -sv http://localhost:3000/api/reports?type=daily-sales
       curl -sv -X POST http://localhost:3000/api/orders
  2. Observe the HTTP response code and Location header.

Expected Result:
  HTTP 401 with body: {"error":"Unauthorized"}
  API clients (mobile apps, automated scripts, non-browser consumers) can parse this
  and handle authentication failure programmatically.

Actual Result:
  HTTP 307 Temporary Redirect
  Location: /login?callbackUrl=http%3A%2F%2Flocalhost%3A3000%2Fapi%2Freports%3F...
  The response body is empty. Any non-browser HTTP client receives a redirect to
  an HTML login page and cannot parse the response as JSON.

Evidence:
  curl -sv "http://localhost:3000/api/reports?type=daily-sales" 2>&1 | grep "< HTTP\|< location"
  < HTTP/1.1 307 Temporary Redirect
  < location: /login?type=daily-sales&callbackUrl=http%3A%2F%2Flocalhost%3A3000%2Fapi%2Freports%3Ftype%3Ddaily-sales

  Same behavior confirmed for: POST /api/orders, POST /api/users, PATCH /api/profile,
  PATCH /api/slots/:id/toggle, POST /api/orders/:id/cancel, POST /api/orders/:id/confirm,
  PATCH /api/deliveries/:id/status

Likely Root Cause:
  middleware.ts applies NextAuth's authorized() callback to ALL matched routes, including
  /api/* paths. The callback returns false for unauthenticated requests, which NextAuth
  translates into a redirect to the signIn page (configured as /login) for both page
  routes and API routes alike. No special handling distinguishes API paths from page paths.

Likely Files Involved:
  middleware.ts (applies uniform redirect to all routes)
  auth.config.ts (authorized callback — no /api/ path check)

Recommended Fix:
  In the authorized callback in auth.config.ts, detect API routes by checking whether
  pathname starts with "/api/" (excluding "/api/auth"). For API paths, return a
  NextResponse.json({ error: "Unauthorized" }, { status: 401 }) instead of allowing
  NextAuth to redirect. The lib/auth/api-auth.ts requireRole() helper already returns
  proper 401/403 responses for routes that reach the handler — the fix is to also
  intercept at the middleware level for routes that are blocked before reaching the handler.

Bug ID:             BUG-004
Title:              orderRef sequence uses db.order.count() — race condition causes 500 under concurrent requests
Severity:           Medium
Area:               Orders / Data Integrity
Status:             Open
Environment:        Local QA
Reproducible:       Intermittent (under concurrent load)

Steps to Reproduce:
  1. Simulate two simultaneous order placements (same customer or different customers):
       curl -s -b <session> -X POST .../api/orders -d <valid_order_body> &
       curl -s -b <session> -X POST .../api/orders -d <valid_order_body> &
       wait
  2. Observe that one request may return a 500 Internal Server Error.
  3. Confirm by checking the Order.orderRef @unique constraint in the schema.

Expected Result:
  Every successful order placement returns a unique orderRef.
  If two requests arrive simultaneously, both should succeed with distinct refs.

Actual Result:
  Two concurrent requests both read the same db.order.count() value N, both compute
  "RAX-YYYYMMDD-00N", and both attempt to INSERT with that ref. The second INSERT
  fails with a unique constraint violation. The API returns HTTP 500 to that customer
  with no useful error message. The customer's order is lost without notification.

Evidence:
  Schema constraint confirmed:
    SELECT indexdef FROM pg_indexes WHERE tablename='Order' AND indexname LIKE '%orderRef%';
    CREATE UNIQUE INDEX "Order_orderRef_key" ON public."Order" USING btree ("orderRef")

  Code in app/api/orders/route.ts:
    const orderCount = await db.order.count();
    const orderRef = generateOrderRef(orderCount + 1);
  These two lines are not atomic. Two requests reading simultaneously get the same count.

Likely Root Cause:
  The orderRef generation reads a non-atomic count, computes a ref, then inserts. There
  is no lock, no DB sequence, and no retry logic. The unique constraint is the only safety
  net, but it surfaces as a 500 rather than a clean duplicate-detection and retry.

Likely Files Involved:
  app/api/orders/route.ts (lines ~70-71: db.order.count() then generateOrderRef)
  lib/utils.ts (generateOrderRef — pure function, not the issue)

Recommended Fix:
  Replace the count-based sequence with a collision-resistant approach. Options in
  increasing robustness: (1) append a short random suffix to the ref and retry on unique
  constraint violation; (2) use a PostgreSQL sequence via Prisma ($queryRaw is not in use
  elsewhere — a migration adding a sequence is clean); (3) use a UUID-based ref internally
  and generate a human-readable ref as a separate presentation concern. At minimum, catch
  the P2002 Prisma unique constraint error in the catch block of route.ts and return a
  422 with a meaningful message rather than a 500.

Title:              Seeded loyalty counts do not match qualifying completed orders in the database
Severity:           Medium
Area:               Loyalty / Data Integrity

Reproducible:       Always (data state issue, not runtime)

Steps to Reproduce:
  1. Run the following reconciliation query:

              COUNT(o.id) as actual_qualifying
       FROM "Customer" c
       JOIN "User" u ON u.id=c."userId"
       LEFT JOIN "Order" o ON o."customerId"=c.id
         AND o.status='COMPLETED'
         AND o."countsForLoyalty"=true
         AND o."isLoyaltyReward"=false
       GROUP BY u.name, c."loyaltyCount";

Expected Result:
  loyaltyCount matches the count of completed qualifying orders in the database for
  every customer. Per business-rules.md §11.1: "Only completed and valid qualifying
  orders shall count toward loyalty."

Actual Result:
  John Njoroge: stored=3, actual qualifying=0
  Jane Kamau:   stored=7, actual qualifying=1 (RAX-20260411-001)
  Mary Achieng: stored=0, actual qualifying=0

Evidence:
      name     | stored | actual_qualifying
  John Njoroge |      3 |                 0
  Jane Kamau   |      7 |                 1
  Mary Achieng |      0 |                 0

  Jane's stored count increases from 7 to 8 correctly when Order 7 is completed
  (loyalty increment was tested and verified). The discrepancy is in the seed data.

Likely Root Cause:
  prisma/seed.ts pre-sets loyaltyCount values (7 for jane.kamau, 3 for john.njoroge)
  to simulate prior off-system order history, but does not seed the corresponding
  COMPLETED Order rows. This produces a divergence between the stored count and any
  loyalty audit based on DB records. The runtime increment logic is correct. This is
  a seed data design issue, not a runtime code bug.

Likely Files Involved:
  prisma/seed.ts (loyaltyCount: 7 and loyaltyCount: 3 set without matching COMPLETED rows)

Recommended Fix:
  Either (a) seed additional COMPLETED Order rows for jane.kamau and john.njoroge that
  account for the pre-set loyalty counts (7 and 3 qualifying orders respectively), giving
  the demo database a consistent state for audit queries; or (b) add a comment in seed.ts
  explaining that the counts represent off-platform history and are intentionally not
  backed by Order rows in this seed. Option (a) is recommended as it makes the demo more
  robust and prevents false positives in any loyalty audit tooling.
